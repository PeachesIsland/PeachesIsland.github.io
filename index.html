Session的一种方式。

GET请求和POST请求的区别是什么？

  1. 使用场景 GET用来获取资源 POST用来传输实体主体

http是什么？

发表于 2023-07-05 分类于 http

http是什么？

http 是超文本传输协议 ，是一个应用层的协议

主要特点：

  1. 支持客户/服务器模式
  2. 简单快速
  3. 灵活
  4. 无连接，http1.0 使用短链接，http1.1（目前主流） 起默认使用长连接（keep-alive）。
  5. 无状态

http请求结构有哪些？

  1. 请求行（Request Line），包含请求方法、请求的URL和HTTP协议的版本。
  2. 请求头（Request Headers）包含一系列的键值对，用于描述请求的各种属性和要求。常见的请求头包括：
  • Host：指定服务器的域名或IP地址。
  • User-Agent：标识发送请求的用户代理（通常是浏览器）。
  • Accept：指定可接受的响应内容的MIME类型。
  • Content-Type：指定请求体的MIME类型（仅适用于POST、PUT等包含请求体的方法）。
  • Authorization：用于身份验证的凭证信息
  1. 空行：请求头和请求体之间必须有一个空行。
  2. 请求体（Request Body）：可选项，包含发送给服务器的数据，通常用于POST、PUT等需要提交数据的请求方法

http响应报文有哪些？

  1. 状态行（Status Line） 包含HTTP协议版本、状态码和对应的状态信息，示例：HTTP/1.1 200 OK
  2. 响应头（Response Headers）
  3. 空行：响应头和响应体之间必须有一个空行。
  4. 响应体（Response Body）

常见http状态码有哪些？

200：ok，正常返回信息

400：客户端请求语法错误

403：重定向

404：客户端请求的资源不存在

500：服务器错误

503：服务器错误

什么情况下服务器会出现大量close_wait,该如何解决？

发表于 2023-07-04 更新于 2023-07-05 分类于 http

什么情况下服务器会出现大量 close_wait,该如何解决？

基于tcp四次挥手的过程，close_wait 会出现在第二次挥手。
如果是服务器出现大量 close_wait，那么主动发起第一次挥手的就是客户端。

这种情况一般是程序忘记执行第三次挥手，没有及时释放导致的，一般就是程序有bug

解决方式：

  1. 检查代码，特别是释放资源的代码
  2. 检查配置，特别是处理请求的线程配置

什么情况下服务器会出现大量time_wait,该如何解决？

发表于 2023-07-04 更新于 2023-07-05 分类于 http

什么情况下服务器会出现大量 time_wait,该如何解决？

基于tcp四次挥手的过程，time_wait 会出现在第四次挥手。
如果是服务器出现大量 time_wait，那么主动发起第一次挥手的就是服务端。
主动方要处于 time_wait 状态。

time_wait是正常的tcp连接断开的步骤，在等待2msl，2msl一般情况下是分钟或几分钟的时间，根据操作系统的设置而定。
到时间之后会进入CLOSED状态，即断开连接。
在短时间内服务器会出现大量 time_wait的话可能会导致服务器资源耗尽，这通常发生在短时间内建立大量的连接并快速关闭它们的情况下。

解决方式：
1 调整操作系统的TCP参数：可以通过调整操作系统的TCP连接超时时间和端口重用等参数来减少TIME_WAIT状态的持续时间。
2 使用连接复用
3 调整应用程序逻辑

关闭处于TIME_WAIT状态的连接是由操作系统自动处理的，应用程序通常不需要显式地关闭这些连接。
TIME_WAIT状态是TCP协议的一部分，旨在保证可靠的连接关闭和数据传输。

什么是tcp四次挥手，为什么需要四次？

发表于 2023-07-04 更新于 2023-07-05 分类于 http

四次挥手

  1. 挥手是为了终止连接
  2. 客户端或服务端都可主动发送挥手信号
  3. TCP连接必须经过2MSL后才能真正释放

以下为客户端主动发起终止请求：

第一次挥手（FIN）：客户端 —->（发送带有FIN标志的TCP报文） —-> 服务端； （客户端进入FIN_WAIT_1状态）

第二次挥手（ACK）：服务端 —->（发送带有ACK标志的TCP报文） —-> 客户端；（服务端进入CLOSE_WAIT状态，客户端进入FIN_WAIT_2状态）

第三次挥手（FIN）：服务端 —->（发送带有FIN标志的TCP报文段） —-> 客户端；（客户端进入LAST_ACK状态）

第四次挥手（ACK）：客户端 —->（发送带有ACK标志的TCP报文） —-> 服务端；（客户端进入TIME_WAIT状态，等待一段时间后[时间：2MSL]进入CLOSED状态，服务端接收ACK后直接进入CLOSED状态）

注意：

CLOSE_WAIT：只会出现在第二次挥手后，被动接收放会进入 CLOSE_WAIT 状态

TIME_WAIT：只会出现在第四次挥手后，主动发起方会进入 TIME_WAIT 状态

time_wait 等待时间为什么是 2MSL ？

MSL：最大报文段生存时间（Maximum Segment Lifetime）。
MSL是指一个TCP报文在网络中的最长生存时间，它的值根据具体的实现和网络环境而定，一般被认为是2分钟（120秒）。
原因：
1 确保有足够的时间让对方收到ack包
2 避免新旧连接混淆

小林coding提到：
在 Linux 系统里 2MSL 默认是 60 秒，那么一个 MSL 也就是 30 秒。Linux 系统停留在 TIME_WAIT 的时间为固定的 60 秒。

为什么需要四次挥手才能断开连接？

因为是全双工的，发送方和接收方都需要FIN报文和ACK报文。
发送方和接收放都各自需要两次挥手。
一方主动，一方被动，就看起来是四次挥手了。
